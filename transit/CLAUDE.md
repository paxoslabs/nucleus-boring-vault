## Architecture: server-approved quotes (current)
The contract holds **no fee or route configuration**. Instead the PXL backend authenticates
integrators by API key, computes fees + the offer→want rate dynamically, and signs a `Quote`.
`submitOrder` verifies that the quote was signed by the trusted `quoteSigner` (a single,
owner-settable address), enforces an immutable `MAX_PROTOCOL_FEE_BPS` cap on the protocol fee, **checks the route against an
on-chain global allowlist**, collects funds, and dispatches. This deletes the old *fee* config
and the per-enterprise policy system (RouteConfig, signerPolicies, routeConfigForPolicy,
_calcFees, setPolicyRoutes, setSignerPolicies, DEFAULT_POLICY) — but **keeps a global on-chain
route allowlist** (`approvedRoutes`).

**Route allowlist (re-added 2026-06-02; receive-side check REMOVED 2026-06-11).** `mapping(uint32 destEID => mapping(address offerAsset => mapping(address wantAsset => bool))) approvedRoutes` —
global per station, NOT per enterprise. Set via `setRouteApprovals(Route[], bool[])`. Enforced **once, at
submission on the source** (`_verifyAndCollect`), so a compromised backend can only ever land orders for
globally-approved asset pairs. `_lzReceive` does NOT re-check: reaching it with an unapproved route requires
defeating LZ peer authentication, and an attacker with that power would forge *amounts* on an approved route
instead — either way value-out stays gated by the executor + exact per-batch approvals + pause. Ops win:
a cross-chain route needs approval only on the SOURCE station ({destEID=remote, offer, want}); no message can
strand on a destination config mismatch. **No merkle root** — a mapping gives O(1) white/blacklist +
on-chain getters.

**Residual risk to remember (UPDATED 2026-06-10 — the big `amountDue` drain is CLOSED):** the rate is no
longer signer-controlled. `amountDue` is **derived on-chain** from the collected post-fee value (18-decimal
normalized, truncated into want units in `_pushOrder` — KDD 27), so a compromised `quoteSigner` + EXECUTOR
can no longer over-price an order and drain `wantAssetSource` — the want delivered always reflects what was
actually deposited (modulo fees + truncation, which always rounds against the receiver). What a compromised
signer CAN still do: grief fees (`protocolFee` ≤ `MAX_PROTOCOL_FEE_BPS` 0.5%; `integratorFee` ≤
`MAX_INTEGRATOR_FEE_BPS` 10% — KDD 28, so total skim ≤ 10.5% per deposit it gets signed) and mis-route within
the owner-approved set. The 1:1 assumption itself is an
**owner/multisig trust** (only approve genuinely like-valued pegs; the vault eats a depeg). `quoteSigner`
rotatable via `setQuoteSigner`; `pause()` freezes submit + execute.

## First-draft contract shape (TransitStation.sol)
Inheritance: `OAppAuth, Pausable`. `Auth` is initialized directly in the constructor
(`Auth(_owner, _authority)`) because `OAppAuthCore` inherits `Auth` but does **not** call
its constructor — the teller relies on its other branch (`TellerWithMultiAssetSupport`) to do
it; we have no such branch. `Pausable` (`src/helper/Pausable.sol`) is the modified-OZ variant
(idempotent `_pause`) relocated out of `one-to-one-queue/access/` (cursed path).

Constructor: `(address _owner, Authority _authority, address _endpoint, address _protocolFeeRecipient, address _quoteSigner, address _offerReceiver, address _wantAssetSource)`. Zero-address checks on all of `_owner`/`_protocolFeeRecipient`/`_quoteSigner`/`_offerReceiver`/`_wantAssetSource`; code-existence check on `_authority` (interface dep). `thisChainEID` is fetched from `endpoint.eid()` rather than passed in.

Structs:
- `Route { uint32 destEID; address offerAsset; address wantAsset; }`
- `OrderTerms { bytes32 uuid; address wantAsset; address receiver; address offerAsset; uint256 offerAmountNormalized18AfterFees; }` — everything fixed at submit time on the source; **this is the LZ wire format** (nothing bridges empty). `offerAmountNormalized18AfterFees` is re-normalized FROM the token units actually collected (never the signed figure). `uuid` is the quote's **EIP-712 digest** (see UUID below). No `sourceEID` (removed 2026-06-11): on the source it's the chain the log lives on; on the destination `OrderBridgeReceived` carries the LZ-authenticated `origin.srcEid` — no sense bridging a copy.
- `Order { OrderTerms terms; uint256 amountDue; uint64 queuedAt; }` — **composition, not field duplication** (2026-06-10): the source-attested terms plus the two fields only the destination can fill. **`_pushOrder(OrderTerms)` is the only place an `Order` is constructed or queued** — it derives `amountDue = _toTokenDecimals(terms.offerAmountNormalized18AfterFees, wantAsset.decimals())` and stamps `queuedAt` at construction, so a half-initialized `Order` (placeholder zeros) never exists anywhere, and wire data can never smuggle an `amountDue` (the wire only carries terms).
- (No `PeerChain` struct — pruned 2026-06-03. Per-EID LZ gas is a single `mapping(uint32 => uint64) messageGasLimit`.)
- `Quote { Route route; uint256 offerAmountNormalized18; address receiver; uint256 protocolFeeNormalized18; uint256 integratorFeeNormalized18; address integratorFeeReceiver; uint256 deadline; bytes32 salt; }` — backend-signed. **ALL amounts normalized to 18 decimals** (field names carry the unit so the backend can't misread them as raw); truncated down to the offer asset's native units at transfer time. **`amountDue` was REMOVED (2026-06-10)** — the signer no longer states the amount delivered; it's derived on-chain (see Order). Two fees: `protocolFeeNormalized18` → `protocolFeeRecipient` (capped at `MAX_PROTOCOL_FEE_BPS`), `integratorFeeNormalized18` → `integratorFeeReceiver` (capped at `MAX_INTEGRATOR_FEE_BPS` — KDD 28). `distributorCode` (bytes32) is a backend-attested referral tag — pure attribution, no funds logic; emitted (indexed — and as a value type the topic carries the code itself) on `OrderSubmitted`, never bridged or stored (KDD 29). **Bearer** — no `payer` binding; anyone can submit (they pay the offer amount, want goes to the fixed `receiver`, so a griefer only burns their own funds).

Storage:
- `mapping(uint32 => uint64) messageGasLimit` — per-destination-EID LZ executor gas for the peer's lzReceive. (Per-chain allowlisting + peer authenticity now come entirely from LZ's own `peers` mapping + `setPeer`; route/asset gating from `approvedRoutes`. The old `PeerChain` allowFrom/allowTo/peerStation were all redundant — pruned.)
- `address quoteSigner` — the single trusted backend signer (owner-settable via `setQuoteSigner`).
- `address protocolFeeRecipient` — receives `protocolFee` (PXL's cut). The integrator's cut goes to the per-quote `integratorFeeReceiver` instead, so they're now separate on-chain (revises KDD 21's single-recipient decision).
- `address offerReceiver` — where the net deposit (`offerAmount - protocolFee - integratorFee`) is sent on submit; not held by the station (owner-settable via `setOfferReceiver`). In practice a BoringVault, but the contract only ERC20-transfers to it — role-named, not enforced as a vault.
- `address wantAssetSource` — the address `executePendingOrders` pulls the want asset FROM (via `transferFrom`) to the receiver; must approve the station (owner-settable via `setWantAssetSource`). KDD 26: the station custodies nothing — assets flow through, never rest.
- `mapping(uint32 => mapping(address => mapping(address => bool))) approvedRoutes` — global per-station directional route allowlist (`destEID => offerAsset => wantAsset`). Nested (not a routeHash) so callers can read `approvedRoutes(eid, offer, want)` directly.
- `mapping(bytes32 => bool) usedDigests` — replay protection.
- `EnumerableSet.Bytes32Set pendingOrderIds` + `mapping(bytes32 => Order) pendingOrders` (per KDD 2)

Constants:
- `ONE_HUNDRED_PERCENT = 10_000` (basis-points denominator)
- `MAX_PROTOCOL_FEE_BPS = 50` (0.5%) and `MAX_INTEGRATOR_FEE_BPS = 1000` (10%) — hardcoded immutable caps on the two fees (KDD 28); together they bound what a compromised `quoteSigner` can skim per signed deposit. Not owner-settable by design.

Functions (meat only):
- `submitOrder(Quote quote, bytes signature) payable requiresAuth whenNotPaused returns (bytes32 uuid)` — public entrypoint, but `requiresAuth` (boring-vault convention: gate everything; the deploy script opens it to the world via `rolesAuthority.setPublicCapability(submitOrder.selector, true)` — gives a revoke kill-switch beyond `pause`). `_verifyAndCollect` checks deadline (`QuoteExpired`), **route is in `approvedRoutes`** (else `RouteNotApproved`), `receiver != 0` (`ZeroAddress`), `integratorFeeReceiver != 0` when the integrator fee > 0 (`ZeroAddress`), both fee caps (`ProtocolFeeTooHigh` / `IntegratorFeeTooHigh`) and `fees < offerAmount` (`FeesExceedOffer`, now defense-in-depth + zero-offer guard) in normalized space, recovers the signer from the **EIP-712** digest (`ECDSA.recover`) and requires `== quoteSigner` (`InvalidSigner(recovered)`), marks `usedDigests[digest]` (`SignatureAlreadyUsed`), then truncates each normalized amount to the offer asset's token units (`NetTruncatesToZero` if the net collects to zero) and pulls from msg.sender as an **exact partition** of the truncated offer amount: protocol fee → `protocolFeeRecipient`, integrator fee → `integratorFeeReceiver`, net → `offerReceiver` (sub-token-unit fee dust folds into the net; station never custodies the offer asset). `_verifyAndCollect` **returns `(digest, offerAmountNormalized18AfterFees)`** — the digest is the `uuid`; the second value (re-normalized from the COLLECTED token units) rides in the Order, and `_pushOrder` derives `amountDue` from it (`ZeroAmountDue` checked there, post-derivation). Then dispatches local-or-LZ. **Every order requires a valid quote — no unsigned/DEFAULT path.**
- `submitOrderWithPermit(...) payable requiresAuth whenNotPaused` — same as `submitOrder` but first runs EIP-2612 `permit(msg.sender, address(this), <truncated token-unit offer amount>, ...)` so the user approves + submits in one tx (the permit covers exactly what the partitioned transfers pull). `try/catch` falls back to an allowance check (`PermitFailedAndAllowanceTooLow`). Shared core in internal `_submitOrder` (avoids an external self-call that would clobber `msg.sender`). Also opened via `setPublicCapability` at deploy.
- `setRouteApprovals(Route[] routes, bool[] approved) requiresAuth` — batched, length-checked global route white/blacklist.
- `executePendingOrders(FillBatch[] batches) requiresAuth whenNotPaused` — EXECUTOR fulfills by `safeTransferFrom`-ing wantAsset from `wantAssetSource` directly to the receiver (station custodies nothing), decrements `amountDue`, removes when zero (partial fills). **Input is grouped by want asset** (2026-06-11): `FillBatch { address wantAsset; bytes32[] uuids; uint256[] amounts; }` — each order's `terms.wantAsset` must equal its batch's `wantAsset` (else `WantAssetMismatch(uuid, orderWantAsset, batchWantAsset)`), which replaces the old on-chain O(n²) token dedup; the residual check is one `allowance(wantAssetSource, this) == 0` read per batch (else `ResidualApproval`). The per-batch check also forces all fills for an asset into a single batch. **Now `whenNotPaused`** (2026-06-08) — pausing halts fulfillment, the compromised-backend kill-switch. **Operational cost: the vault must approve EXACTLY the per-asset batch total before each execute, or it reverts.** Reverts everything on any failure with the offending `uuid` (`OrderNotFound` / `AmountExceedsDue` / `WantAssetMismatch`). NOTE: `fillAmount == 0` is not rejected (auth'd executor; harmless no-op) — flagged, left as-is.
- `forceRemovePendingOrder(bytes32 uuid) requiresAuth` — force-remove, full only.
- `recoverETH(uint256) requiresAuth` / `recoverTokens(ERC20, uint256) requiresAuth` — to owner.
- `pause() requiresAuth` / `unpause() requiresAuth` — emergency stop via `Pausable` (`src/helper/Pausable.sol`, idempotent `_pause`). **`whenNotPaused` gates `submitOrder`, `submitOrderWithPermit`, AND `executePendingOrders`** (revised 2026-06-08). Execute MUST be pausable: EXECUTOR-gating alone is NOT enough, because the EXECUTOR key itself can be compromised/hacked — pause is the kill-switch that freezes fulfillment (the value-out step) even against a malicious executor, as well as a compromised backend whose bad orders are already queued. (Do NOT regress this to "execute stays operational when paused" — earlier docs/CodeRabbit had that backwards.) `_lzReceive` is intentionally NOT gated: no value moves on receive (only on execute, which is gated), and gating it would strand in-flight cross-chain messages. `paused()` is the public view.
- `setProtocolFeeRecipient/setQuoteSigner/setOfferReceiver/setWantAssetSource(addr) requiresAuth` (all zero-checked; `quoteSigner` has no code check — it's an EOA key), `setMessageGasLimit(eid, uint64) requiresAuth`.
- `_lzReceive(origin, guid, payload, …)` — decodes `OrderTerms` and pushes (`_pushOrder` derives `amountDue` from the bridged normalized net + local `wantAsset.decimals()`, stamps `queuedAt`, and returns the populated copy), then emits `OrderBridgeReceived(uuid, origin.srcEid, guid, order)`. Sender authenticity enforced upstream by `OAppAuthReceiver.lzReceive` (`peers[srcEid]`); the route is **NOT re-checked** (validated once on the source — removed 2026-06-11, see Decisions Log). A revert here (e.g. `ZeroAmountDue`) is safe — delivery is **unordered** (`nextNonce` returns 0), so the message stays retryable on the endpoint; nothing is consumed/lost (see Decisions Log 2026-06-06).
- `quoteSend(destEID, OrderTerms) view` — LZ fee preview (encodes the same payload + options as `_sendOrder`, so the estimate matches the real send).
- Views: `getPendingOrders()` → returns the full `Order[]` (was `getPendingOrderIds` returning `bytes32[]`; renamed 2026-06-06 to materialize each order — **O(n) full copy, off-chain readers only**, an on-chain caller risks OOG), `pendingOrderCount`, plus the public `pendingOrders`/`usedDigests` mappings and the nested `approvedRoutes(eid, offer, want)` auto-getter.

Fee model: two fees (normalized 18-decimal units, like all quote amounts), both computed off-chain and signed into the quote, both capped by hardcoded immutables. `protocolFeeNormalized18` (PXL's cut) → `protocolFeeRecipient`, capped at `MAX_PROTOCOL_FEE_BPS` (0.5%). `integratorFeeNormalized18` (the frontend's own UniswapX-style fee) → the per-quote `integratorFeeReceiver`, capped at `MAX_INTEGRATOR_FEE_BPS` (10% — KDD 28; the fee/receiver are backend-signed, so a backend compromise alone could otherwise set any fee to any receiver). The net is pulled to `offerReceiver` (in practice a BoringVault) in token units, and the **want owed is derived from the collected net** (re-normalized, then truncated into want units on the destination) rather than taken from the quote.

UUID generation (revised 2026-06-08): the `uuid` **is the quote's EIP-712 digest** (`keccak256(0x1901 ‖ domainSeparator ‖ hashStruct(Quote))`), returned by `_verifyAndCollect`. Replaced the old nonce scheme `keccak256(thisChainEID, address(this), user, ++orderNonce)` — `_newUuid` and `orderNonce` deleted. Rationale: the nonce made the UUID depend on submission *order* and *submitter*; the digest is fully deterministic from the signed quote (backend knows it at sign time), order/submitter-independent (matters since quotes are bearer), and cross-chain-unique because the domain separator binds `block.chainid` + station `address(this)`. The `salt` field disambiguates otherwise-identical quotes. `usedDigests[digest]` and `pendingOrders[digest]` are now keyed identically — the digest *is* the order identity, so a consumed quote's UUID can never be reused (resubmit reverts on `usedDigests` before recreating it). Also ~18k gas cheaper/submit (no nonce SSTORE).

Events (for off-chain tracking; the two LZ events were added 2026-06-06):
- `OrderSubmitted(uuid, sourceEID, route, terms, user, distributorCode)` — fully collected + dispatched (local or bridged). `sourceEID` = `thisChainEID` (kept as an event param after the struct field was removed, so consumers get it without an EID-per-chain lookup). Carries `OrderTerms` — identical/honest on both paths (no placeholder zeros); the queued state is in `OrderReceived` for same-chain orders and only exists on the destination for bridged ones. `distributorCode` is **indexed** and `bytes32`, so the topic carries the code itself — filterable AND readable straight from the log (KDD 29).
- `OrderBridged(uuid, destEID, guid, terms)` — source side of a cross-chain dispatch; carries the **exact bridged payload** (`OrderTerms`) + the LZ `guid` (trace it / correlate with the destination's `OrderBridgeReceived`).
- `OrderReceived(uuid, order)` — order entered a pending set (fires on BOTH local submit and cross-chain receive); the executor's canonical "now fillable" signal.
- `OrderBridgeReceived(uuid, srcEID, guid, order)` — destination side of a bridged order; full stamped order + matching `guid`. (Intentional redundancy with `OrderReceived` on the cross-chain path: different consumers — executor vs. ops/correlation.)
- `OrderExecuted(uuid, amount, remaining)` — per fill (`remaining == 0` ⟺ fully filled).
- `OrderForceRemoved`, plus the setter events.
- `_pushOrder` now `returns (Order memory)` (the stamped copy) so `_lzReceive` can log the stored state in `OrderBridgeReceived`.

## What We're Building (One-Paragraph Summary)
**Transit Stations** are per-chain smart contracts that let a user deposit one asset (e.g., USDC) and request a different asset on the same or a different chain (e.g., USDG on Ethereum). The actual swap is orchestrated off-chain by a privileged executor. On submission, the user presents a `Quote` signed by the PXL backend (which authenticated their integrator API key, picked the route, and priced fees + rate). The station verifies the signature against `quoteSigner`, enforces `MAX_PROTOCOL_FEE_BPS` on the protocol fee, deducts `protocolFee` → `protocolFeeRecipient` and `integratorFee` → `integratorFeeReceiver`, and either:
- records a pending order locally (single-chain), or
- sends a LayerZero message containing the **full order data** to the destination station (cross-chain).

The destination station's `_lzReceive` validates the route and places the order in an `EnumerableSet` of order UUIDs mapped to `Order` structs. Backend iterates and batch-fulfills, with partial fills supported (decrement `amountDue` until zero, then remove). Fulfillment is permissioned via RolesAuthority `EXECUTOR` — users never pull funds. Contracts are immutable (no upgradeability). No user-initiated refunds, no order expiry, no on-chain max-size (off-chain rate limit guards LZ reorg risk). Owner can force-remove orders and recover stray ETH/ERC20s; vault retains custody/refund authority.

## Cross-Chain Handling — Important Nuance
Cross-chain is **message-passing only, never value-passing**:
1. Source station takes user's deposit + fees locally.
2. Source station emits a LayerZero message to destination station containing the **full order data** (asset, amount, receiver, source chain, etc. — per KDD 3, not just a hash).
3. Destination station's `_lzReceive` validates the route and records the order in its own pending-orders set — as if a single-chain order had been submitted there directly.
4. Executor fulfills by pulling the want asset from the destination station's configured `wantAssetSource` (KDD 26) straight to the receiver via `transferFrom`; the station custodies nothing.

Funds never bridge — the deposit goes to the source `offerReceiver`, and the wanted asset is pulled from the destination station's `wantAssetSource` to the receiver (KDD 26; station holds nothing).

### LZ choice (KDD 4)
LayerZero over CCIP for configurable block confirmations / latency. 3 DVNs assumed for security. Formula for delivery time: `(srcBlockTime × confirmations) + (destBlockTime × (2 + numDVNs))`.

### Translation: Teller → Transit Station
| Teller concept | Transit equivalent |
|---|---|
| `shareAmount` payload | Full `Order` struct (asset, amount, receiver, srcChain, offerAsset, offerAmount, queuedAt, UUID) |
| `vault.enter(..., receiver, shareAmount)` in `_lzReceive` | Push order into `EnumerableSet` + `mapping(bytes32 => Order)` |
| `selectorToChains[].targetTeller` allowlist | Same pattern — only accept messages from the known peer Transit station per EID |
| `BridgeData.messageGas` / `OptionsBuilder` | Same — required because destination payload size is larger, so gas limits matter more |
| `requiresAuth` on `bridge()` | On Transit, `submitOrder` is `requiresAuth` but opened to the public via `setPublicCapability` at deploy (revoke kill-switch); `executePendingOrders` is `requiresAuth` (EXECUTOR role, not public) |

## Key Design Decisions (Quick Reference — see DESIGN.md for full context)
- KDD 1: Per-chain mapping config, **no merkle root**.
- KDD 2: `EnumerableSet<bytes32>` of UUIDs + `mapping(bytes32 => Order)`. **No swap-and-pop** (breaks batch processing).
- KDD 3: Bridge **full** order data, not hashes.
- KDD 4: **LZ** over CCIP.
- KDD 5: **Not upgradeable** — new versions are new deployments.
- KDD 6/7: **No on-chain refunds** (user or protocol-initiated); admin force-remove only.
- KDD 8: **Partial fills allowed** at smart-contract level; min applies per order, not per fill.
- KDD 9: **No on-chain max size**; rate limits live off-chain.
- KDD 10: **Per-route minimum** order size.
- KDD 11: **No expiry**.
- KDD 12: Single-chain **and** cross-chain in the same contract.
- KDD A: **Privileged execution only** (EXECUTOR role).
- KDD B/C/E: **Additive flat + percent fees**, per route.
- KDD D: No partial force-remove.

- *2026-06-10*: **Normalized-18 amounts; `amountDue` derived on-chain (KDD 27).** Closes the headline drain: a compromised `quoteSigner` could previously sign `offerAmount=$1, amountDue=$1B` and (with the same-trust-domain executor) drain `wantAssetSource`. `amountDue` is REMOVED from the signed `Quote`; ALL quote amounts are now **normalized to 18 decimals** (`offerAmountNormalized18`, `protocolFeeNormalized18`, `integratorFeeNormalized18` — names carry the unit) so **token decimals never bridge**. Collection truncates each amount to the offer asset's units (`_toTokenDecimals`) and the three transfers partition the truncated offer exactly (sub-unit fee dust folds into the net → vault; zero-unit net reverts `NetTruncatesToZero`); the Order carries `offerAmountNormalized18AfterFees` **re-normalized from the COLLECTED units** (never the signed figure), and the destination's `_pushOrder` truncates it into want units for `amountDue` (`ZeroAmountDue` post-derivation; truncation always rounds against the receiver, so want delivered ≤ offer collected at 1:1). `submitOrderWithPermit` permits the truncated token-unit amount. *(History: a first approach the same day bridged raw `net` + `offerDecimals` via a `BridgeData` wire struct — implemented, reverted, superseded by normalization, which needs neither.)* New EIP-712 `QUOTE_TYPEHASH`; backend must sign normalized amounts; testnet stations are pre-change bytecode. Tests extended to 8 (decimal scaling both directions, sub-dust reverts both guards, fee-dust partition). *(The then-open `MAX_INTEGRATOR_FEE_BPS` item was closed the same day — see the KDD 28 entry below.)*
- *2026-06-10*: **`OrderTerms` composition — wire format = the terms, `Order` embeds them.** Follow-up to the above: the full `Order` was still the LZ payload, bridging `amountDue`/`queuedAt` as always-zero words. The LZ fee saving is negligible (~64 zero bytes ≈ 1–2k gas equivalent vs. the 400k requested gas limit — the real bridge-cost lever is right-sizing `messageGasLimit`), so the change was made for **type honesty**, not fees: `OrderTerms {uuid, wantAsset, receiver, sourceEID, offerAsset, offerAmountNormalized18AfterFees}` holds everything fixed at submit time and is what bridges; `Order {OrderTerms terms, amountDue, queuedAt}` composes it with the destination-derived state (no field duplication, no `_toOrder` copier — unlike the reverted `BridgeData`, which modeled the wire as "Order minus fields"). `_pushOrder(OrderTerms)` is the sole constructor of `Order` values (derives `amountDue` + stamps `queuedAt` at construction — callers never pass placeholder zeros; Carson: "things should only be passed as they are"); `OrderSubmitted`/`OrderBridged` now carry `OrderTerms` (honest on both paths); `quoteSend` takes `OrderTerms` so estimates match the real payload. ABI note: `pendingOrders`/`getPendingOrders`/order events now return/carry the nested struct — off-chain decoders see `(terms tuple, amountDue, queuedAt)`.
- *2026-06-10*: **Integrator fee cap (KDD 28).** `MAX_INTEGRATOR_FEE_BPS = 1000` (10%), hardcoded immutable like the protocol cap, enforced in `_verifyAndCollect`. Carson's rationale: the uncapped fee assumed the *frontend* was the threat (a phished frontend already fully exposes the user, so a cap adds nothing there) — but the fee + `integratorFeeReceiver` are **backend-signed**, so a backend hack alone could set any fee to any receiver and "essentially steal from any user"; a user could catch a changed fee/receiver pre-submit, "but we don't want to put that responsibility on them." `FeeTooHigh` split into `ProtocolFeeTooHigh`/`IntegratorFeeTooHigh` so reverts identify which cap tripped. Combined bound: a full backend compromise skims ≤ 10.5% per signed deposit; `FeesExceedOffer` stays as defense-in-depth + the zero-offer guard. (Initially implemented at 1%; Carson raised it to 10% to leave room for partner pricing.)
- *2026-06-10*: **Distributor code (KDD 29).** Added `distributorCode` to the signed `Quote` — an arbitrary referral tag so enterprises can log flow as theirs ("this flow was mine"; pure attribution, distinct from the integrator fee — no funds move). After `DistributorCodeDepositor`'s referral-code idea, emitted as an indexed field on `OrderSubmitted`. Design calls: **signed into the quote** (backend authenticates the integrator, so attribution is backend-attested — a tampered code fails signature verification and changes the UUID) and **source-only** (not in `OrderTerms`, never bridged or stored — attribution is a source-chain fact; cross-chain joins by `uuid`). **Revised 2026-06-11 from `bytes` to `bytes32`** (Carson): an indexed dynamic type puts only `keccak256(value)` in the topic, so consumers couldn't recover the code from the log — and EIP-712 reduces `bytes` to its 32-byte hash for signing anyway. As a `bytes32` value type, the indexed topic carries the code itself (filterable AND readable), `_hashQuote` encodes it atomically (no `keccak256` wrapper), and it costs one calldata word. `bytes32(0)` = unattributed. Test note (for when the suite returns): solc 0.8.21 ICEs on `emit TransitStation.OrderSubmitted(...)` (qualified access to a NatSpec'd event — fixed in 0.8.22), so tests need a local mirror event for `vm.expectEmit`.
- *2026-06-11*: **Removed `sourceEID` from `OrderTerms`.** It was only ever read by events. On the destination, `OrderBridgeReceived` already carries `origin.srcEid` — the LZ-authenticated value, strictly more trustworthy than a sender-written payload field. One less word bridged/stored. `OrderSubmitted` gained a `sourceEID` param (= `thisChainEID`) so source-side consumers still get the EID directly from the log rather than needing a chain→EID mapping.
- *2026-06-11*: **`executePendingOrders` takes `FillBatch[]` — fills grouped by want asset** (coworker's idea via Carson). Replaces `(bytes32[] uuids, uint256[] amounts)` + the on-chain O(n²) `usedTokens` dedup: the backend groups fills per asset (`FillBatch { wantAsset, uuids, amounts }`), the contract asserts each order's `terms.wantAsset` equals its batch's (`WantAssetMismatch(uuid, orderWantAsset, batchWantAsset)`), and the custody/residual check becomes one `allowance == 0` read per batch. The per-batch residual check also implicitly forces all fills for an asset into one batch (a split would leave allowance > 0 after the first and revert). Supersedes the 2026-06-05 "self-derivation over a backend-passed token list" decision — the grouped structure is both cheaper and simpler than the dedup it replaced. Decoder note: `NucleusDecoderAndSanitizer.executePendingOrders` still mirrors the OLD signature — realign before any merkle root is built against the new selector.
- *2026-06-11*: **Removed the route re-check in `_lzReceive`** (supersedes "enforced on BOTH source and destination" in the 2026-06-02 entry, and the 2026-06-08 deployment note that routes must be approved on both stations — **routes now need approval only on the SOURCE station**). Carson's rationale: the check is only reachable past LZ peer authentication, so exploiting receive requires an LZ-level attack — and that attacker's obvious lever is *amounts*, not the route; even a forged route is bounded by the station's pullable inventory, and the executor still has to choose to fill. Checking only on send gives the same security surface with less confusion/complexity. Verified before removal: `executePendingOrders` never re-checks routes, so the receive check never gated value-out — it only filtered delivery; the value-out gates (EXECUTOR role, exact per-batch approvals, `pause`) are unchanged. Trade-offs accepted: (a) de-approving a route no longer blocks in-flight bridged orders for it (it never blocked queued ones either — pause/executor are the kill switches); (b) `RouteNotApproved` stranded-message recovery flow is gone because the strand can't happen. `_isRouteApproved` now has a single caller (`_verifyAndCollect`).
